Passivbot includes a multi-objective evolutionary optimizer that searches for configurations on the Pareto front — the set of candidates where no single objective can be improved without sacrificing another. The optimizer evaluates candidates through historical backtests and uses either the `pymoo` (default, NSGA-II/III) or `deap` backend. Optimization requires the full install profile.

<Info>
  Install the full profile before running optimization:

  ```bash
  pip install -e ".[full]"
  ```
</Info>

## Running optimization

```bash
passivbot optimize [path/to/config.json]
```

If no config is specified, the optimizer starts from the in-code schema defaults in `src/config/schema.py`. The example config `configs/examples/default_trailing_grid_long_npos10.json` mirrors those defaults exactly.

Useful flags:

- **`--start path/to/config(s)`** — Seed the initial population with existing configs.
- **`--fine_tune_params param_a,param_b`** — Lock all bounds except the listed parameters (short: `-ft`).
- **`--suite [y/n]`** — Enable suite scenarios defined in `backtest.scenarios`.
- **`--suite-config path/to/file.json`** — Layer additional scenario definitions.

Example using a starting pool:

```bash
passivbot optimize configs/examples/default_trailing_grid_long_npos10.json \
  --start configs/starting_pool/
```

For a full list of CLI arguments:

```bash
passivbot optimize -h
```

## Backend selection

Passivbot supports two optimizer backends, configured via `optimize.backend`:

<AccordionGroup>
  <Accordion title="pymoo (default)">
    The default backend. Uses NSGA-II for 3 or fewer objectives and NSGA-III for 4 or more, when `algorithm` is set to `"auto"`.

    ```json
    {
      "optimize": {
        "backend": "pymoo",
        "pymoo": {
          "algorithm": "auto"
        }
      }
    }
    ```
  </Accordion>
  <Accordion title="deap">
    The legacy evolutionary backend. Still supported but no longer the default.

    ```json
    {
      "optimize": {
        "backend": "deap"
      }
    }
    ```
  </Accordion>
</AccordionGroup>

## NSGA-III and reference directions

NSGA-III is a many-objective evolutionary algorithm. Unlike NSGA-II, it distributes candidates across reference directions in objective space rather than relying solely on crowding distance.

Passivbot uses the `das_dennis` method to generate those reference directions. The main knob is `n_partitions`:

| `n_partitions` | Reference directions (8 objectives) |
|----------------|-------------------------------------|
| `3` | 120 |
| `4` | 330 (default when `"auto"`) |
| `5` | 792 |

With the default 8-objective scoring set and `n_partitions = "auto"`, Passivbot resolves to `n_partitions = 4`, giving 330 reference directions. The `population_size` defaults to `null`, which in NSGA-III mode uses the number of reference directions as the population size (330 for the default setup).

Full pymoo configuration reference:

```json
{
  "optimize": {
    "backend": "pymoo",
    "population_size": null,
    "pymoo": {
      "algorithm": "auto",
      "shared": {
        "crossover_eta": 20.0,
        "crossover_prob_var": 0.5,
        "mutation_eta": 20.0,
        "mutation_prob_var": "auto",
        "eliminate_duplicates": true
      },
      "algorithms": {
        "nsga2": {},
        "nsga3": {
          "ref_dirs": {
            "method": "das_dennis",
            "n_partitions": "auto"
          }
        }
      }
    }
  }
}
```

## Optimization bounds

Bounds define the search space for each tunable parameter under `optimize.bounds`. Three formats are supported:

| Format | Description |
|--------|-------------|
| `[low, high]` | Continuous range |
| `[low, high, step]` | Stepped range — use for integer parameters |
| `value` | Fixed value — parameter is excluded from search |

Use the stepped format for integer parameters like `n_positions` to prevent the optimizer from evaluating fractional values:

```json
{
  "optimize": {
    "bounds": {
      "long_n_positions": [1, 20, 1],
      "long_entry_grid_spacing_pct": [0.001, 0.1],
      "long_entry_initial_qty_pct": [0.001, 0.1]
    }
  }
}
```

## Scoring metrics

The `optimize.scoring` list defines the objectives the evolutionary algorithm optimizes. Each entry becomes one fitness component.

Default objectives include metrics from the following families:

- **Returns & growth**: `adg_w`, `adg_strategy_pnl_rebased_w`, `mdg_w`
- **Risk**: `drawdown_worst_hsl`, `drawdown_worst_mean_1pct_hsl`, `drawdown_worst_ema_hsl`
- **Efficiency**: `sharpe_ratio_w`, `calmar_ratio_w`, `omega_ratio_w`
- **Recovery**: `peak_recovery_hours_hsl`

Metrics with the `_w` suffix use recency-weighted means: the metric is evaluated on ten overlapping slices of the equity curve and averaged, biasing the score toward recent behavior without ignoring the past.

All metrics are available with `_usd` and `_btc` denomination suffixes (e.g., `adg_usd`, `adg_btc`). Exposure-normalized variants (e.g., `adg_per_exposure_long`) divide the base metric by the configured `total_wallet_exposure_limit`.

## Optimization limits

Limits define hard constraints. Candidates that violate a limit are penalized rather than disqualified, which pushes them far from the Pareto front.

Each limit object supports:

- `metric` — canonical metric name (e.g., `drawdown_worst_btc`, `loss_profit_ratio`, `adg`)
- `penalize_if` — operator: `<`, `<=`, `>`, `>=`, `==`, `outside_range`, `inside_range`
- `value` — numeric threshold for scalar operators
- `range` — `[low, high]` for range-based operators
- `stat` — override the statistic compared (`min`, `max`, `mean`, `std`); defaults to `_max` for `>` checks and `_min` for `<` checks

```json
{
  "optimize": {
    "limits": [
      {"metric": "drawdown_worst_btc", "penalize_if": ">", "value": 0.3},
      {"metric": "loss_profit_ratio", "penalize_if": "outside_range", "range": [0.05, 0.7]},
      {"metric": "adg", "penalize_if": "<", "value": 0.0008, "stat": "mean"}
    ]
  }
}
```

CLI overrides using the `--limit` flag (keep-condition semantics for scalar operators):

```bash
passivbot optimize \
  --clear-limits \
  --limit 'drawdown_worst <= 0.35' \
  --limit 'backtest_completion_ratio>=1.0' \
  --limit 'loss_profit_ratio outside_range [0.05,0.7]' \
  --limit 'adg > 0.0008 stat=mean'
```

You can also mix `--limits` (full JSON replacement) with `--limit` (appends one entry):

```bash
passivbot optimize \
  --limits '[{"metric":"drawdown_worst","penalize_if":">","value":0.35}]' \
  --limit 'peak_recovery_hours_hsl <= 500'
```

## Fine-tuning specific parameters

When you want to adjust only a handful of parameters and keep everything else fixed, use `--fine_tune_params` (short: `-ft`) with a comma-separated list of `optimize.bounds` keys:

```bash
passivbot optimize configs/examples/default_trailing_grid_long_npos10.json \
  --fine_tune_params long_entry_grid_spacing_pct,long_entry_initial_qty_pct
```

All unlisted bounds are collapsed to `[value, value]`, so the optimizer can only mutate the parameters you specified.

The config-file equivalent is `optimize.fixed_params`: list the `optimize.bounds` keys that should always be fixed to their current config values. Internally, `--fine_tune_params` and `optimize.fixed_params` are merged into one effective fixed-parameter set before bounds are collapsed.

<Note>
  `optimize.fixed_runtime_overrides` is different from `fixed_params`. It overrides runtime config values only during optimizer evaluations without changing the stored config. This is commonly used to disable terminal no-restart during optimization so candidates can be constrained through drawdown and recovery limits instead of being prematurely truncated.
</Note>

## Candle interval

For faster optimization runs, aggregate 1-minute data into coarser candles before the backtest loop:

```json
{
  "backtest": {
    "candle_interval_minutes": 5
  }
}
```

This reduces the number of bars processed per iteration. Intra-interval fill ordering is lost (fills occur only at aggregated bar boundaries), but metrics remain time-correct because analysis uses timestamps rather than bar indices.

## Analyzing results

Two tools are available for exploring the Pareto front after a run.

### Pareto dashboard (recommended)

Launch an interactive Plotly Dash app that scans one or more optimization run directories:

```bash
passivbot tool pareto-dash --data-root optimize_results
```

Pass `--run optimize_results/<timestamp>/` to load a specific run, or point it at the entire `optimize_results/` directory to browse multiple runs at once. The dashboard includes scatter/histogram views, scenario-aware box plots, correlation heat maps, parameter-vs-metric scatter plots, and a streaming history chart from `all_results.bin`.

### Pareto single-candidate selector

Pick one candidate from a large Pareto front without opening the dashboard:

```bash
# Use the newest local pareto directory automatically
passivbot tool pareto

# Use a specific pareto directory
passivbot tool pareto optimize_results/.../pareto

# Select closest to specified targets
passivbot tool pareto optimize_results/.../pareto \
  -m reference \
  --target adg_strategy_pnl_rebased=0.001 \
  --target drawdown_worst_hsl=0.25

# Filter then rank by weighted utility
passivbot tool pareto optimize_results/... \
  -m utility \
  --weight adg_strategy_pnl_rebased=4 \
  --weight drawdown_worst_hsl=2 \
  --show-top 5

# Output JSON for scripting
passivbot tool pareto --json
```

Available selection methods:

| Method | Description |
|--------|-------------|
| `ideal` | Closest to the observed ideal point (default) |
| `knee` | Approximate balanced compromise point |
| `reference` | Closest to user-specified targets via `--target metric=value` |
| `utility` | Highest weighted normalized utility via `--weight metric=value` |
| `lexicographic` | Strict objective priority via `--priority metric_a,metric_b,...` |
| `outranking` | Simplified PROMETHEE-style pairwise net-flow ranking |

Recommended workflow: apply hard filters with `--limit`, use `-m reference` if you have a specific ADG/drawdown target, otherwise start with the default `ideal` selector.

`passivbot tool pareto-explorer` is a CLI alias for the same tool.

## Output structure

Each optimization run creates a directory:

```
optimize_results/YYYY-MM-DDTHH_MM_SS_{exchanges}_{n_days}days_{coin_label}_{hash}/
```

Contents:

| Path | Description |
|------|-------------|
| `all_results.bin` | Binary log of all evaluated configs (msgpack format) |
| `pareto/` | JSON files for Pareto-optimal configurations, named `{hash}.json` |
| `index.json` | List of current Pareto member hashes |

Pareto files are added and removed over time as the front updates and is pruned to `optimize.pareto_max_size`. Suite runs switch the coin portion of the folder name to `suite_{n}_coins` to make them easy to locate.

## Optimizer suites

The optimizer reuses the backtest suite configuration, evaluating every candidate across multiple scenarios before scoring. Each scenario can override coins, date ranges, exchanges, `coin_sources`, and bot parameters via `overrides`. Passivbot prepares a single shared dataset spanning the union of the requested data so additional scenarios add minimal overhead.

Key fields under `backtest`:

- `suite_enabled` — master toggle; can also be set with `--suite [y/n]`
- `scenarios` — list of scenario dictionaries (same schema as backtest scenarios)
- `aggregate` — how to combine per-scenario metrics (default: `{"default": "mean"}`)

Suite mode is opt-in. The default schema and example config do not enable it automatically. See the [Backtesting](/backtesting) guide for suite configuration examples.
